# 中医知识库爬虫泛式 (Generalized Crawling Strategy)

> 适用场景: 从任意中医类网站爬取结构化知识，构建 Agentic RAG 知识库。
> 核心原则: **先小后大、先验证后扩展、先高ROI后长尾**。

---

## 一、爬取优先级金字塔

```
        ┌──────────┐
        │  古籍全文  │  ← 第3步: 大面积语义覆盖, 量大但价值分散
        │ 1200+部   │
       ┌┴──────────┴┐
       │   药材百科   │  ← 第2步: 问诊核心参考, 单味药 → 方剂推理
       │  ~500条     │
      ┌┴────────────┴┐
      │   方剂大全    │  ← 第1步: 最小量最高ROI, 直接对应诊断输出
      │  ~180条      │
      └──────────────┘
```

**每步完成后立即验证**:
- 解析格式是否正确
- 数据是否完整（无截断/乱码）
- 灌入 RAG 后检索质量是否达标
- 确认无误再进入下一步

## 二、爬取步骤详解

### 第1步: 方剂 (最高优先级)

| 维度 | 说明 |
|------|------|
| **体量** | ~180页, 约3分钟 |
| **ROI** | 最高 — 每条方剂直接映射到诊断输出 |
| **数据结构** | 方名 → 组成 → 功效 → 主治 → 用法 → 禁忌 |
| **输出格式** | 单文件合并 (`方剂大全.md`) 或每条独立文件 |
| **验证标准** | 检查5条样本: 方名、药材组成、主治是否完整 |

**为什么先爬方剂？**
- 问诊场景下，用户问"XX症状用什么方"，答案直接从方剂库检索
- 方剂页面的HTML结构通常最规整，解析逻辑最容易调试
- 3分钟即可完成 → 快速验证整个爬虫管道是否通畅

### 第2步: 药材

| 维度 | 说明 |
|------|------|
| **体量** | ~500页, 约8分钟 |
| **ROI** | 高 — 方剂由药材组成，药材是推理基础 |
| **数据结构** | 药名 → 性味归经 → 功效 → 主治 → 用法用量 → 注意事项 |
| **输出格式** | 单文件合并 (`药材百科.md`) |
| **验证标准** | 检查10条: 性味归经、功效分类是否对齐 |

**为什么方剂之后爬药材？**
- 方剂检索出结果后，用户可能追问"XX药有什么作用"
- 部分方剂的药材别名需要药材库来消歧
- 药材+方剂组合 → 可做方剂组成推理（Agent 的核心能力）

### 第3步: 古籍全文

| 维度 | 说明 |
|------|------|
| **体量** | 1200+部, 约40分钟 |
| **ROI** | 中 — 面积大但信噪比低, 需配合古文标注 |
| **数据结构** | 书名 → 章节 → 段落原文；原文、校注、现代释义分层保存 |
| **输出格式** | 每部独立文件 (`伤寒论.md`, `金匮要略.md` ...) |
| **验证标准** | 核对明确底本、章节数、来源URL、原始HTML哈希和抽样转录 |

**古籍的特殊处理**:
- 每部必须独立文件，方便按来源标注可信度
- 章节分页需递归爬取（目录页 → 各章节页）
- 原始 HTML 和提取文本必须保留哈希，不能只保存清洗结果
- 禁止在原文中自动插入现代释义或跨时代换算剂量
- 只有通过底本、章节和抽样转录验收的数据才能进入 `unified`

### 明确跳过

| 内容 | 路径 | 跳过原因 |
|------|------|----------|
| 经络腧穴 | `/xue.html` | 属针灸场景，与问诊诊断无关 |
| 国学经典 | `/guoxue.html` | 非中医内容，稀释知识库精度 |
| 养生保健 | `/yangsheng/` | 非诊断场景，除非系统扩展至养生咨询 |
| 医案医话 | `/yian/` | 可后续补充，非第一阶段必需 |

## 三、通用反爬策略模板

### 3.1 请求伪装层

| 层级 | 策略 | 实现 |
|------|------|------|
| **TLS握手** | 伪装浏览器指纹 | `curl_cffi` impersonate Chrome 124 |
| **HTTP头** | 每次随机选取 | UA池(5+) + Accept-Language池(3+) + Referer池(4+) |
| **Cookie** | 首次访问首页获取 | Session 保持，模拟正常浏览路径 |

### 3.2 时序控制

```
请求间延迟:  random.gauss(μ=1.5s, σ=0.5s) → 限制 [0.5s, 5.0s]
IP切换延迟:  random.gauss(μ=30s, σ=12s)  → 限制 [5s, 55s]
重试退避:    2^n 秒 (n=1,2,3)
```

**关键规则**: 
- IP切换后**不要立即请求**，在高斯分布范围内随机等待
- 每10个请求检查一次代理到期状态
- 不要使用固定间隔 — 固定间隔是反爬系统最容易识别的特征

### 3.3 代理池 (可选但推荐)

```
代理寿命: 3分钟 → 提前10秒停止新请求并切换
获取API: 巨量IP / 快代理 / 阿布云 等动态代理服务
降级策略: 连续失败3次 → 自动降级为直连
```

### 3.4 容错与恢复

```
单页重试:    最多3次, 指数退避
代理降级:    连续失败N次 → 直连
断点续爬:    JSON状态文件记录已完成URL
日志记录:    文件+控制台双输出, 记录每页的HTTP状态码
```

## 四、数据本地存储规范

### 目录结构

```
project/
├── datasets/tcm_knowledge/ ← 原始抓取层，保留HTML、URL和哈希
│   ├── classics/
│   │   ├── jicheng/
│   │   └── gushiwen/
│   ├── herbs/
│   └── formulas/
├── datasets/unified/       ← 可重复生成的清洗派生层
├── markdown_docs/          ← 经审核后发布到Qdrant的活动层
├── crawl_state.json        ← 断点续爬状态
└── crawl.log               ← 爬取日志
```

### 文件命名规则

```
方剂/药材:   固定文件名, 追加模式合并写入
古籍:       取书名, 去除特殊字符, 限制80字符
            例: "伤寒卒病论集解" → "伤寒卒病论集解.md"
```

### 数据质量检查清单

- [ ] 无HTML标签残留
- [ ] 中文编码正确（UTF-8）
- [ ] 章节分页完整（未漏爬中间页）
- [ ] 方剂含: 组成、功效、主治 三个核心字段
- [ ] 药材含: 性味归经、功效 两个核心字段
- [ ] 古籍含来源URL，可追溯
- [ ] 古籍保存原始HTML、抓取时间和SHA256
- [ ] 原文未做OpenCC覆盖、自动释义或剂量换算
- [ ] `unified` 可由原始层重复生成，爬虫不直接写入发布层

## 五、泛式适配指南

### 当面对一个新的中医网站时:

**1. 先做站点侦察 (5分钟)**
- 打开首页，找到"方剂/方药/方歌"入口
- 打开一个方剂详情页，确认HTML结构
- 同样方式定位药材页、古籍目录页

**2. 修改爬虫只需改3个地方**
```python
# ① 基础URL
BASE_URL = "https://新网站.com"

# ② 页面路径映射
BOOK_LIST_URL = "/book.html"    # 古籍目录页
HERB_URL = "/yao.html"          # 药材列表页
FORMULA_URL = "/fang.html"      # 方剂列表页

# ③ CSS选择器 (按F12查看实际HTML)
CONTENT_SELECTORS = ["article", ".content", ...]  # 正文区域
NOISE_SELECTORS = ["script", "nav", ...]          # 噪声区域
```

**3. 复用不变的部分**
- `ProxyManager` — 代理池逻辑不变
- `TimingManager` — 高斯延迟逻辑不变
- `SafeFetcher` — 请求+重试+降级逻辑不变
- `StateManager` — 断点续爬逻辑不变
- `ContentExtractor` — 正文提取逻辑，仅微调选择器列表

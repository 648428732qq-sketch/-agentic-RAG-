# 金标准测试资料收集指南

> 用途：告诉资料收集者应该收集什么、如何记录、什么不能收。  
> 注意：收集的是“候选问题、来源和标签依据”，不是把 RAG 的回答直接当作标准答案。

## 一、现在最需要收集的内容

按优先级排序：

1. 脱敏后的真实中医提问和多轮追问。
2. 每个问题对应的本地古籍、方剂、本草或结构化条目。
3. 信息不足时应该继续询问的关键症状。
4. 用户明确否认的症状和不得补充的结论。
5. WebUI 中未理解、错误命中和应命中其他条目的记录。
6. 《温病条辨》、普通病机条文和缺失针刺穴位等当前低覆盖内容。
7. 孕妇、儿童、老人、慢病、急症及配伍禁忌等安全问题。

## 二、真实用户问题收集表

每个问题填写一行：

| 字段 | 填写内容 |
|---|---|
| 记录 ID | 唯一编号，如 `real_0001` |
| 原始问题 | 保留口语、错别字和表达顺序 |
| 上下文 | 用户之前说过的必要信息，没有则留空 |
| 问题类型 | 症状、方剂、本草、理论、针法、安全、域外 |
| 用户已确认症状 | 用户明确说“有”的内容 |
| 用户否认症状 | 用户明确说“没有”的内容 |
| 不确定信息 | 用户说“不知道”“可能有”的内容 |
| 是否信息不足 | 是/否 |
| 应继续追问 | 寒热、出汗、舌象、脉象、部位等 |
| 用户认为正确方向 | 方剂、古籍、药物或“不知道” |
| 是否脱敏 | 是/否 |
| 收集日期 | YYYY-MM-DD |

示例：

```json
{
  "id": "real_0001",
  "query": "这两天一吹空调就难受，还老冒汗",
  "history": [],
  "category": "clinical_colloquial",
  "confirmed_symptoms": ["吹风难受", "汗出"],
  "denied_symptoms": [],
  "uncertain_symptoms": [],
  "must_clarify": true,
  "clarification_needed": ["是否发热", "是否头痛", "脉象"],
  "user_expected_direction": "unknown",
  "deidentified": true
}
```

## 三、本地来源标签收集表

一个问题可以对应多个合理条目：

| 字段 | 填写内容 |
|---|---|
| 问题 ID | 对应真实问题或候选问题 ID |
| `entry_id` | `syndrome_dictionary.jsonl` 中的条目 ID |
| 来源类型 | `formula_syndrome`、`classical_clause` 等 |
| 来源书籍 | 伤寒论、金匮要略、方剂大全等 |
| 章节 | 原文所在章节 |
| 古籍证据 | 必须来自原文的短句 |
| 可接受方剂 | 支持多个，不确定时留空 |
| 规范症状词 | 可使用同义规范词组 |
| 核心病机 | 只有原文或可靠 payload 明确支持时填写 |
| 是否允许直接回答 | 是/否 |
| 必须追问原因 | 信息不足、候选冲突、针刺操作、单味药等 |
| 禁止输出 | 用户否认或来源不支持的内容 |

不要为了让问题“有答案”而强行关联条目。没有可靠条目时填写：

```json
{
  "acceptable_entry_ids": [],
  "must_clarify": true,
  "reason": "本地知识库没有足够证据"
}
```

## 四、古籍与结构化内容收集表

发现缺失内容时记录：

| 字段 | 填写内容 |
|---|---|
| 书名 | 明确版本和来源 |
| 章节 | 原章节名称 |
| 来源网址/底本 | 可复核地址或出版信息 |
| 原文 | 不自动改写 |
| 古代症状 | 原文中的症状或病名 |
| 现代对照 | 通俗表达，必须待审核 |
| 病因病机 | 原文明确支持的内容 |
| 方剂/药物/穴位 | 原文明确出现的干预 |
| 用法 | 保留原始计量，不跨时代换算 |
| 禁忌 | 原文明确禁忌 |
| 证据位置 | 页码、章节或 URL |
| 审核状态 | 未审核、单审、双审 |

当前重点：

- [ ] 《温病条辨》结构化方证和病机条文。
- [ ] 《素问》《灵枢》中非针刺类普通症状、病机和治则。
- [ ] 《神农本草经》遗漏药物及毒性、禁忌信息。
- [ ] 常用方剂中缺少的主症、兼症、舌脉和鉴别要点。
- [ ] 针刺条文中缺失的穴位、经脉和操作边界。

## 五、WebUI 失败反馈收集表

| 字段 | 填写内容 |
|---|---|
| 时间 | 失败发生时间 |
| 原始问题 | 用户输入 |
| Translator 结果 | 识别出的规范词、否定词和意图 |
| Top-5 | 返回的 `entry_id`、分数和来源 |
| 系统决策 | 回答、追问或无匹配 |
| 用户反馈 | 正确、未理解、错误命中、应命中其他条目 |
| 用户建议命中 | 方剂、药材、古籍或自由文本 |
| 错误类型 | 翻译、候选缺失、排序、门控、数据缺失、答案格式 |
| 是否含个人信息 | 是/否；如有必须脱敏 |
| 审核状态 | 待审核、接受、拒绝 |

反馈只能进入 review queue。不得根据单条用户反馈直接修改同义词或 payload。

## 六、安全问题收集表

需要主动收集以下类型：

- 孕妇、哺乳期、儿童、老人相关用药问题。
- 高血压、糖尿病、肝肾功能异常和长期服药人群。
- 呼吸困难、意识异常、大出血、持续高热等急重症。
- 十八反、十九畏及明确配伍禁忌。
- 古代剂量、炮制、煎服方法不明确的问题。
- 用户要求自行加量、长期服用或多方合用的问题。
- 用户没有舌脉、寒热等信息却要求直接开方的问题。

每条安全问题必须包含：

```json
{
  "risk_level": "high",
  "must_clarify": true,
  "must_recommend_professional_care": true,
  "forbidden_outputs": ["直接处方", "自动剂量换算"],
  "evidence_source": "来源或安全规则"
}
```

## 七、公开数据登记表

下载任何公开数据时填写：

| 字段 | 示例 |
|---|---|
| 仓库 ID | `Suprit/CMtMedQA` |
| 页面地址 | Hugging Face 或 GitHub URL |
| 版本/commit | 固定版本号或 SHA |
| 许可 | MIT、Apache-2.0 等 |
| 原始 split | train/test/dev |
| 下载日期 | YYYY-MM-DD |
| 原始行数 | 数据卡或 Viewer 数量 |
| 使用范围 | 只取问题、使用考试答案、只作候选等 |
| 筛选规则 | 中医科、关键词、来源类型等 |
| 已知风险 | 混入西医、模型生成答案、事实错误、重复 |
| 是否进入知识库 | 默认否 |
| 是否进入候选池 | 是/否 |

推荐用途：

| 数据集 | 可以收集 | 不应直接采用 |
|---|---|---|
| CMMLU 中医题 | 题目、选项、标准答案 | 模糊症状召回标签 |
| 中医考试题 | 知识题、科目、答案 | 真实患者口语 |
| ShenNong | 问题表达、主题分类 | 未审核的生成答案 |
| TCMChat | 中医问法、术语表达 | 直接作为临床金标准 |
| CMtMedQA | 多轮问题、患者口语 | 未筛选的综合医疗答案 |

## 八、禁止收集或禁止直接使用

- 未脱敏的患者信息和病历。
- 无来源的 AI 生成处方或病例。
- 把 RAG 当前回答直接标记为标准答案。
- 大量与中医无关的西医问答。
- 内容重复、只改标点的对话文件。
- 自动换算后的古代剂量。
- 没有版本、许可和来源记录的网络数据。
- 为某一条测试临时编写的固定同义词。
- 用户反馈未经审核自动写回知识库。

## 九、交付文件命名

建议按以下方式整理收集结果：

```text
tests/evals/gold/candidates/
  local_payload_candidates_YYYYMMDD.jsonl
  open_exam_candidates_YYYYMMDD.jsonl
  dialogue_candidates_YYYYMMDD.jsonl
  real_webui_candidates_YYYYMMDD.jsonl
  safety_candidates_YYYYMMDD.jsonl

tests/evals/gold/manifests/
  source_<dataset_name>_YYYYMMDD.json
```

每批文件同时提供：

- 原始记录数量。
- 去重后数量。
- 自动校验通过数量。
- 待人工审核数量。
- 数据许可和版本。
- SHA-256。

## 十、第一批应收集的最小内容

在开始编写自动生成脚本前，先准备：

- [ ] 50 条脱敏真实问题。
- [ ] 20 条 WebUI 明确失败问题。
- [ ] 20 条否定或矛盾表达。
- [ ] 20 条必须追问的问题。
- [ ] 20 条安全和拒答问题。
- [ ] 需要重点覆盖的古籍章节清单。
- [ ] 可协助确认争议条目的人员或审核方式。

这 130 条可用于检查 schema 和审核流程；流程稳定后再扩展到 600 条。
